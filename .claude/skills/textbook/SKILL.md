# textbook — 学習テキストの章/分野を統一フォーマットで作る

このリポジトリは、複数分野（Audio・強化学習・LLM・Physical AI …）を**同じ構成・同じ品質**で
書く学習テキスト。コンテンツを追加・編集するときは、必ずこの手順に従い、品質と再現性を保つ。

## 最初に必ず読むもの（毎回）

1. **`AUTHORING.md`** — 品質基準・トーン・数式/コード規約・手順の唯一の正典。**最優先で全文読む。**
   とくに **§2「説明の深さ基準」** が品質の心臓部（記号は全定義／構造でなく動作／誤解先回り／1ステップ例＋エッジケース／学習時vs推論時／why／対比・アナロジー）。
2. **`templates/chapter-template.md`** — 章の雛形（見出しと順序）。
3. **`docs/audio/01-digital-audio-basics.md`** — 完成章のお手本。粒度・密度・数式とコードの書き方をここに合わせる。
4. **`zensical.toml`** — `nav`（ナビゲーション）の現状を把握する。

これらと食い違う指示がこのファイルにあれば、`AUTHORING.md` を優先する。

## モード A: 新しい章を書く

1. **対象分野と位置を確認する。** どの分野（`docs/<domain>/`）の、何番目の章か。番号 `NN` と slug を決める。
   分野の `index.md` のロードマップ・章一覧と整合させる。曖昧なら、何の章をどの深さで書くかをユーザーに 1 回だけ確認する。
2. **下調べ。** 内容の正確さが命。必要なら WebSearch / WebFetch で原典・定義・式を確認する。**数式は導出を自分で追って検算する。コードは実際に実行して出力を確かめ、章に載せる出力はその実測値にする**（憶測で出力を書かない）。numpy は導入済みなので `uv run python ...` で動く。他のライブラリ（torch 等）が要るときは `uv run --with <pkg> python ...` で一時的に持ち込む。
3. **執筆。** `templates/chapter-template.md` をベースに `docs/<domain>/NN-<slug>.md` を作る。§2 の深さ基準を全項目満たす。
   - 順序固定: 学習目標 → 前提知識 → 直感 → **全体像** → 理論 → 数式の導出 → 実装 → 演習 → まとめ → **用語ミニ辞典** → **次のアクション** → 参考文献。
   - 全体像では順方向・逆方向を 1 枚で一望させる（Mermaid 第一選択。Mermaid で描けない領域固有の図は Canvas: 章には `<canvas class="tb-fig" id="...">` だけ置き、描画関数は `docs/javascripts/figures.js` のレジストリに登録。§7 の制約内）。
   - 誤解しやすい点（添字の衝突など）は `!!! warning` で名指しして否定する。
   - 数式は `$...$` / `$$...$$`、導出の終わりに `$\blacksquare$`。コードは `title=` 付き＋**実測**出力併記、そのまま実行可能に。
   - 使ってよい機能は `AUTHORING.md` §8 に限定（`:material-...:` 等のアイコンショートコードは使わない）。配色・装飾は `extra.css` のトークンに従う。
4. **配線を更新する。**
   - `zensical.toml` の `nav` の該当分野セクションに `{ "N. <章タイトル>" = "<domain>/NN-<slug>.md" },` を追加。
   - `docs/<domain>/index.md` の章一覧テーブルを ✅ に更新し、章へのリンクを張る。前後の章があれば「次に学ぶこと」のリンクも繋ぐ。
5. **検証する。** `uv run zensical build` がエラー・警告なく通ることを確認。`AUTHORING.md` §12 のチェックリストを自己点検する。

## モード B: 新しい分野を追加する

1. ユーザーに分野名・カバー範囲・章のロードマップ案を 1 回確認する（既に十分な指定があれば省略）。
2. `docs/<domain>/`（英小文字ハイフン区切り）を作り、`templates/domain-index-template.md` を `index.md` にコピーして埋める（ロードマップの mermaid と章一覧を含む）。
3. `zensical.toml` の `nav` にトップレベルセクションを 1 つ追加（`index.md` だけでよい）。
4. トップページ `docs/index.md` の「分野一覧」カードに 1 枚追加する。
5. `uv run zensical build` で確認。最初の章を書くかはユーザーに確認、または続けてモード A へ。

## モード C: 既存の章を直す

- 指定箇所のみを surgical に直す。隣接箇所を勝手に「改善」しない。
- ただし `AUTHORING.md` の規約・構成順序からの逸脱を見つけたら、直す範囲に含めてよい。
- 直したら `uv run zensical build` で確認する。

## モード D: 質問された箇所を厚くする（最重要ループ）

ユーザーが既存章について「ここが分からない」「なぜ〜なのか」と質問したら、それは**説明が薄かったシグナル**。チャットで答えるだけで終わらせない。

1. まずチャットで的確に答える。
2. **その理解しづらかった点を、該当章の本文に書き戻して厚くする**（`AUTHORING.md` §2 の深さ基準を、つまずいた一点に集中して適用）。具体例・図（Mermaid/Canvas）・誤解の先回り（`!!! warning`）・動作の順序などを足す。
3. surgical に。該当箇所だけを厚くし、無関係な改変はしない。
4. `uv run zensical build` で確認し、必要なら commit/push でデプロイ。

## 完了の定義

- ビルドが通る（`uv run zensical build`）。
- `AUTHORING.md` §12 のチェックリストを満たす（とくに §2 の深さ基準）。
- `nav` と分野 `index.md` が新しい内容と整合している。
- 数式は検算済み、コードは実行して出力を確認済み。

## やらないこと

- `zensical.toml` を更新せずに新しい Markdown 拡張・依存ライブラリを使う。
- お手本（audio 第1章）より明らかに薄い/雑な章を「完成」とする。
- ユーザーが指定していない分野や章を勝手に量産する。1 回に頼まれた範囲だけ作る。
