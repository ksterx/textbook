# textbook — 学習テキストの章/分野を統一フォーマットで作る

このリポジトリは、複数分野（Audio・強化学習・LLM・Physical AI …）を**同じ構成・同じ品質**で
書く学習テキスト。**Astro** 製（自前レイアウト・KaTeX ビルド時数式・Pagefind 検索・GitHub Pages 公開）。
コンテンツを追加・編集するときは、必ずこの手順に従い、品質と再現性を保つ。

## 最初に必ず読むもの（毎回）

1. **`AUTHORING.md`** — 品質基準・トーン・数式/コード規約・手順の唯一の正典。**最優先で全文読む。**
   とくに **§2「説明の深さ基準」** が品質の心臓部（記号は全定義／構造でなく動作／誤解先回り／1ステップ例＋エッジケース／学習時vs推論時／why／対比・アナロジー）。
2. **`templates/chapter-template.md`** — 章の雛形（見出しと順序）。
3. **`src/pages/audio/01-digital-audio-basics.md`** — 完成章のお手本。粒度・密度・数式とコードの書き方をここに合わせる。
4. **`src/lib/nav.mjs`** — ナビゲーションの現状を把握する。

これらと食い違う指示がこのファイルにあれば、`AUTHORING.md` を優先する。

## スタックの要点（Astro 固有）

- コンテンツは `src/pages/<domain>/NN-<slug>.md`（URL は `/<domain>/NN-<slug>/`）。
- admonition は **`:::` ディレクティブ**で書く（`!!!` ではない）:
  `:::note[タイトル]` … `:::` / `:::warning[…]` / `:::abstract[…]` / `:::success[…]` / `:::question[…]` / `:::tip[…]`。
  折りたたみ（演習の解答など）は `:::details[解答]` … `:::`。演習は外4コロン・内3コロンでネスト:
  `::::question[演習 1: …]` … `:::details[解答]` … `:::` … `::::`。
- 数式は `$...$` / `$$...$$`（KaTeX がビルド時にレンダー。CDN なし）。
- 図は ```mermaid フェンス（自動で描画）／領域固有の図は Canvas（`public/figures.js` に描画関数を登録し、章には `<canvas id="...">` だけ置く）／2次元の並びは `.tb-tgrid` などの CSS グリッド。
- 内部リンクはルート絶対で書く（base は自動付与）: `[x](/audio/02-frequency-and-features/)`。
- 配色・装飾は `src/styles/global.css` のトークンに従う。
- ビルド/確認は **`npm run build`**（`npm run dev` でローカルプレビュー）。

## モード A: 新しい章を書く

1. **対象分野と位置を確認する。** どの分野（`src/pages/<domain>/`）の、何番目の章か。番号 `NN` と slug を決める。
   分野の `index.md` のロードマップ・章一覧と整合させる。曖昧なら、何の章をどの深さで書くかをユーザーに 1 回だけ確認する。
2. **下調べ。** 内容の正確さが命。必要なら WebSearch / WebFetch で原典・定義・式を確認する。**数式は導出を自分で追って検算する。コードは実際に実行して出力を確かめ、章に載せる出力はその実測値にする**（憶測で出力を書かない）。Python は `uv run --with numpy python ...`（必要なら `--with torch` 等）で実行できる。
3. **執筆。** `templates/chapter-template.md` をベースに `src/pages/<domain>/NN-<slug>.md` を作る。§2 の深さ基準を全項目満たす。
   - 順序固定: 学習目標 → 前提知識 → 直感 → **全体像** → 理論 → 数式の導出 → 実装 → 演習 → まとめ → **用語ミニ辞典** → **次のアクション** → 参考文献。
   - frontmatter は `layout: ../../layouts/Page.astro` ／ `title:` ／ `description:`。
   - 全体像では順方向・逆方向を 1 枚で一望（Mermaid 第一選択。描けない領域固有の図は Canvas → `public/figures.js` に登録）。
   - 誤解しやすい点（添字の衝突など）は `:::warning[…]` で名指しして否定する。
   - 数式は `$...$` / `$$...$$`、導出の終わりに `$\blacksquare$`。コードは `title=` 付き＋**実測**出力併記、そのまま実行可能に。
   - 使ってよい機能は `AUTHORING.md` §8 に限定。
4. **配線を更新する。**
   - `src/lib/nav.mjs` の該当分野の `items` に `{ title: 'N. <章タイトル>', href: '/<domain>/NN-<slug>/' }` を追加。
   - `src/pages/<domain>/index.md` の章一覧を ✅ に更新し、章へのリンクを張る。前後の章があれば「次に学ぶこと」のリンクも繋ぐ。
5. **検証する。** `npm run build` がエラーなく通ることを確認。`AUTHORING.md` §12 のチェックリストを自己点検する。

## モード B: 新しい分野を追加する

1. ユーザーに分野名・カバー範囲・章のロードマップ案を 1 回確認する（既に十分な指定があれば省略）。
2. `src/pages/<domain>/`（英小文字ハイフン区切り）を作り、`templates/domain-index-template.md` を `index.md` にコピーして埋める。
3. `src/lib/nav.mjs` にトップレベルのセクションを 1 つ追加。
4. トップページ `src/pages/index.md` の「分野一覧」カードに 1 枚追加する。
5. `npm run build` で確認。最初の章を書くかはユーザーに確認、または続けてモード A へ。

## モード C: 既存の章を直す

- 指定箇所のみを surgical に直す。隣接箇所を勝手に「改善」しない。
- ただし `AUTHORING.md` の規約・構成順序からの逸脱を見つけたら、直す範囲に含めてよい。
- 直したら `npm run build` で確認する。

## モード D: 質問された箇所を厚くする（最重要ループ）

ユーザーが既存章について「ここが分からない」「なぜ〜なのか」と質問したら、それは**説明が薄かったシグナル**。チャットで答えるだけで終わらせない。

1. まずチャットで的確に答える。
2. **その理解しづらかった点を、該当章の本文に書き戻して厚くする**（`AUTHORING.md` §2 の深さ基準を、つまずいた一点に集中して適用）。具体例・図（Mermaid/Canvas）・誤解の先回り（`:::warning`）・動作の順序などを足す。
3. surgical に。該当箇所だけを厚くし、無関係な改変はしない。
4. `npm run build` で確認し、必要なら commit/push でデプロイ。

## 完了の定義

- ビルドが通る（`npm run build`）。
- `AUTHORING.md` §12 のチェックリストを満たす（とくに §2 の深さ基準）。
- `src/lib/nav.mjs` と分野 `index.md` が新しい内容と整合している。
- 数式は検算済み、コードは実行して出力を確認済み。

## やらないこと

- 新しい依存ライブラリや構文を、`package.json`/`astro.config.mjs` を更新せずに使う。
- お手本（audio 第1章）より明らかに薄い/雑な章を「完成」とする。
- ユーザーが指定していない分野や章を勝手に量産する。1 回に頼まれた範囲だけ作る。
- 公開サイトに外部資料の画像を無断で埋め込む（著作権。自作の Mermaid/Canvas/HTML を既定とする）。
